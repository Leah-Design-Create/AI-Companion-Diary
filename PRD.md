## 1. 产品概述

### 1.1 产品定位

**树洞日记**是一款面向个人用户的 AI 情感陪伴产品。用户通过文字或语音与 AI 角色「小伴」进行情感对话，每次对话结束后自动生成配有 AI 插画的日记条目。

产品定位为**情感陪伴工具**，不是心理咨询平台，不是效率工具，是"一个随时在线、有记忆、不评判的朋友"。

### 1.2 产品愿景

> 让每个人在需要倾诉的时候，都有一个懂自己的地方。

### 1.3 核心体验链路

```
打开应用
  → 与小伴聊天（文字 / 语音）
  → 小伴记住你说过的事
  → 结束对话
  → 自动生成日记 + 专属心情画
  → 日记本留存，随时回看
```

### 1.4 差异化优势

| 维度 | 树洞日记 | 通用 AI 助手 | 心理咨询 App |
|---|---|---|---|
| 陪伴感 | ★★★★★ | ★★★ | ★★★ |
| 跨会话记忆 | ★★★★★ | ★★ | ★★ |
| 日记留存 | ★★★★★ | ✗ | ✗ |
| 专业知识 | ★★★（RAG）| ★★★★ | ★★★★★ |
| 隐私性 | ★★★★（本地部署）| ★★ | ★★ |
| 使用门槛 | 极低 | 低 | 高 |

## 2. 商业目标与成功指标

### 2.1 商业目标（当前阶段：MVP 验证）

1. 验证"AI 作为朋友式陪伴"的用户留存价值
2. 沉淀高质量对话数据，用于 AI 人设迭代
3. 建立可复制的多用户私有化部署方案

### 2.2 关键成功指标（KPIs）

#### 用户活跃度

| 指标 | 定义 | 目标值 |
|---|---|---|
| DAU/MAU | 日活 / 月活比 | > 30%（高黏性产品基准）|
| 平均对话轮次 | 每次 session 的来回消息数 | > 6 轮 |
| 会话完成率 | 用户主动点击「结束对话」的比例 | > 40% |
| D7 留存 | 首次使用后第 7 天仍活跃 | > 25% |
| D30 留存 | 首次使用后第 30 天仍活跃 | > 15% |

#### AI 质量（LLM-as-Judge 评分）

| 指标 | 定义 | 目标值 |
|---|---|---|
| 共情分（empathy）| 0-10 | > 7.0 |
| 自然度（naturalness）| 0-10 | > 7.5 |
| 综合分（overall）| 0-10 | > 7.0 |

#### 功能使用率

| 指标 | 定义 | 目标值 |
|---|---|---|
| 日记生成率 | 有总结的会话 / 全部会话 | > 60% |
| 心情画生成率 | 有图的会话 / 全部结束会话 | > 80% |
| 语音使用率 | 使用过 PTT 的用户 / 全部用户 | > 20% |
| 长期记忆命中率 | 召回 > 0 条记忆的请求 / 全部请求 | > 50%（活跃用户）|

### 2.3 反向指标（需避免）

- 用户反馈"感觉像在跟客服聊" → 自然度分跌破 6.0
- AI 每次回复以问句收尾 → 用户产生压力感
- 生成的图片风格与情绪不符 → 用户主动反馈图片差

## 3. 目标用户

### 3.1 用户画像

#### Persona A：独居职场青年「晓雯」

- **年龄**：25-32 岁，城市独居
- **状态**：工作压力大，朋友圈维系成本高，不想"打扰别人"
- **痛点**：有情绪时找不到合适的倾诉对象；说了之后担心被评判
- **期望**：有个地方可以随时说，不被评判，被认真听
- **使用场景**：下班回家、睡前、压力爆发时
- **关键功能**：文字对话、日记留存、长期记忆

#### Persona B：内向敏感型用户「小磊」

- **年龄**：18-25 岁，学生或刚工作
- **状态**：社交焦虑倾向，不擅长表达，情绪波动较大
- **痛点**：说话前需要大量时间整理思路；在乎别人的反应
- **期望**：一个完全不评判的空间；能帮助梳理情绪
- **使用场景**：考试压力、人际冲突后、情绪低落时
- **关键功能**：语音 PTT、情绪分析、心情画

#### Persona C：有记录习惯的用户「Coco」

- **年龄**：28-40 岁，有写日记习惯
- **状态**：将对话视为高质量日记工具
- **痛点**：普通日记 App 太被动，无法对话；AI 没有记忆
- **期望**：每天聊一聊，自动生成可回顾的记录；AI 能记住之前说的事
- **使用场景**：每日复盘、年终回顾
- **关键功能**：日记列表、日历、数据导出、长期记忆

### 3.2 用户旅程地图

#### 首次使用旅程

```
发现产品 → 注册账号 → 进入聊天
→ 第一次说一段话 → 收到小伴回复（感受：被接住了？还是像客服？）
→ 继续聊几轮 → 结束对话
→ 看到生成的日记 + 心情画 → 感受惊喜
→ 次日打开 → 小伴提到昨天说过的事 → 感受"它记得我"
```

#### 流失风险节点

| 节点 | 风险 | 应对 |
|---|---|---|
| 第一条回复 | 像客服、套话、不自然 | 人设严格规范 + 评估监控 |
| 第 3-5 轮 | AI 一直以问句结尾，用户疲惫 | 明确禁止每次问句收尾 |
| 第二天回来 | AI 不记得昨天说的事 | 长期记忆跨会话召回 |
| 查看日记 | 总结太官方、心情画和情绪无关 | 总结 prompt 优化 + 图片质量监控 |

### 5.2 账号与身份系统

#### 业务背景
支持多用户独立使用，每个用户的对话、记忆、日记完全隔离。

#### 功能需求

| ID | 功能 | 优先级 | 描述 |
|---|---|---|---|
| AUTH-01 | 邮箱注册 | P0 | 邮箱 + 密码注册；密码最少 6 位；bcrypt 不可逆加密存储 |
| AUTH-02 | 登录 | P0 | 邮箱 + 密码验证；成功返回 64 位随机 Token，30 天有效 |
| AUTH-03 | 登出 | P0 | 服务端立即删除 Token |
| AUTH-04 | 修改密码 | P1 | 需先验证当前密码 |
| AUTH-05 | 注销账号 | P1 | 级联删除：会话、消息、Token、ChromaDB 向量、上传图片 |
| AUTH-06 | 旧数据接管 | P1 | 首位注册用户若存在 user_id=1 的无邮箱匿名记录，自动接管以保留历史数据 |

#### 鉴权方式
- Header：`Authorization: Bearer <token>`
- 前端存储：`localStorage`
- 未登录 / Token 过期：返回 HTTP 401

#### 验收标准
- [ ] 同一邮箱重复注册返回 409
- [ ] 密码错误返回 401，不暴露具体原因
- [ ] Token 过期后请求返回 401，前端跳转登录页
- [ ] 注销后所有历史数据不可检索

### 5.3 AI 对话（文字）

#### 业务背景
产品核心功能。用户通过文字与小伴进行情感对话。

#### 功能需求

| ID | 功能 | 优先级 | 描述 |
|---|---|---|---|
| CHAT-01 | 多轮对话 | P0 | 在同一 session 内持续对话，保留历史上下文 |
| CHAT-02 | 新建会话 | P0 | 不传 session_id 则自动创建新会话 |
| CHAT-03 | 流式回复 | P0 | SSE 流式，首 Token < 2s，逐字展示 |
| CHAT-04 | 消息拆分 | P0 | 回复拆成 2-3 条短消息，模拟真人连发感 |
| CHAT-05 | 天气融入 | P1 | 前端获取天气后附在请求中，AI 自然融入对话 |
| CHAT-06 | 追问跟进 | P1 | 新会话时检查 2-14 天前是否有未跟进话题（如面试、等结果）|
| CHAT-07 | 上下文截断 | P0 | 仅保留最近 12 条消息；更早内容用 session summary 补充 |

#### 消息拆分逻辑

1. 模型输出含 `[NEXT]` 标记 → 按标记拆分（最多 3 条）
2. 无标记 → 按句末标点自动分句，合并为 2-3 块（每块 ≤ 55 字）
3. 存库时清除 `[NEXT]`，仅存纯文本

#### 验收标准
- [ ] 发送消息后流式回复首字 < 2s
- [ ] 回复被拆为多条按序展示，有气泡间隔感
- [ ] 切换到旧会话后可继续对话，历史消息正确加载
- [ ] 上下文超过 12 条时，仍能正确回复（不忘记当前话题）

### 5.4 AI 对话（语音 PTT）

#### 业务背景
提供语音输入方式，同时通过声音能量分析感知用户情绪状态，让 AI 的回应更有温度。

#### 功能需求

| ID | 功能 | 优先级 | 描述 |
|---|---|---|---|
| PTT-01 | 按住说话 | P0 | 按住 PTT 按钮录音，松开后发送 |
| PTT-02 | 实时语音识别 | P0 | Web Speech API continuous 模式，实时转文字展示 |
| PTT-03 | 多段累积 | P1 | 识别中断后自动重启，使用闭包变量累积全部文字 |
| PTT-04 | 声音能量采集 | P1 | Web Audio API，AnalyserNode，每 100ms 采样一次 RMS |
| PTT-05 | voice_hint 生成 | P1 | 松手时根据 RMS 数据生成声音特征描述，随消息发给 AI |
| PTT-06 | TTS 自动播报 | P0 | AI 回复后自动调用 TTS 播放，通话模式专属 |

#### voice_hint 生成规则

| 维度 | 阈值 | 描述词 |
|---|---|---|
| 音量 | 平均 RMS < 0.04 | 声音较轻 |
| 音量 | 平均 RMS > 0.15 | 声音较响 |
| 音量稳定性 | 标准差高 | 音量有起伏 |
| 内容长度 | 字数 < 20 | 说话简短 |
| 内容长度 | 字数 > 80 | 说了很多 |
| 停顿 | 0值帧占比高 | 有明显停顿 |

示例输出：`"声音较轻，语速偏慢，有明显停顿"`

voice_hint 注入 system prompt 为 `【语音状态参考】`，AI 据此调整语气，**不直接告知用户**。

#### 验收标准
- [ ] PTT 按下即开始识别，识别文字实时展示
- [ ] 松手后 1-2s 内发送（等待最终 finalResult）
- [ ] voice_hint 在服务端日志中可见（`[VoiceHint] "..."` 格式）
- [ ] AI 回复后自动播放 TTS

### 5.6 知识库 RAG

#### 业务背景
管理员可上传心理/情绪相关书籍、文章。聊天时系统自动引用，提升回复的专业性。**用户无感知，引用内容自然融入回复。**

#### 功能需求

| ID | 功能 | 优先级 | 描述 |
|---|---|---|---|
| RAG-01 | 文件上传 | P0 | 支持 .txt / .pdf，上限 20MB |
| RAG-02 | 自动分块 | P0 | 400字/块，50字重叠，优先在段落/句子边界分割 |
| RAG-03 | Embedding 生成 | P1 | 每个 chunk 生成向量，配置 OPENAI_EMBEDDING_MODEL 后启用 |
| RAG-04 | 语义检索 | P1 | 余弦相似度 ≥ 0.45 则注入，最多 6 条片段 |
| RAG-05 | 关键词检索 | P0 | jieba 分词 + LIKE 查询，命中 ≥ 1 则注入（兜底）|
| RAG-06 | 触发条件 | P0 | 含知识类关键词（焦虑/情绪/心理等）→ 强制调用；其余由模型决定 |
| RAG-07 | 跳过条件 | P0 | 追问词（"然后呢""继续"等）或消息 ≤ 2 字 → 跳过 |
| RAG-08 | 片段截取 | P0 | 围绕关键词位置截取，而非简单取文档开头 |

#### 验收标准
- [ ] 上传 PDF 后，提问相关内容时 AI 能引用其中信息
- [ ] "然后呢"类追问不触发 RAG（服务端日志确认）
- [ ] 未配置 embedding 时关键词检索正常工作
- [ ] 上传超出大小限制时返回友好错误

### 5.7 长期记忆

#### 业务背景
用户最核心的差异化体验：小伴记得你。跨会话记住用户说过的事，实现"有记忆的朋友"感。

#### 功能需求

| ID | 功能 | 优先级 | 描述 |
|---|---|---|---|
| MEM-01 | 对话对写入 | P0 | 每轮对话后台写入 ChromaDB，格式：`用户：xxx\n小伴：xxx` |
| MEM-02 | 用户画像提取 | P0 | LLM 后台提取姓名/偏好/经历/情绪，写为 profile 条目 |
| MEM-03 | 跨会话检索 | P0 | 每次发消息前检索 top-4 相关历史片段，注入 system |
| MEM-04 | 排除当前会话 | P0 | 只召回其他会话的记忆，避免重复注入当前对话内容 |
| MEM-05 | 姓名优先 | P1 | 用户问"你记得我叫什么"时，直接查 profile_name 条目，置顶返回 |
| MEM-06 | 用户消息优先 | P1 | 用户发言的 distance score 额外 -0.08，提升召回率 |
| MEM-07 | ChromaDB 可选 | P1 | 未安装 chromadb 时自动跳过，不影响核心对话 |

#### 后台并行处理（不阻塞响应）

每轮对话结束后，同时异步执行：
1. `add_turn_to_memory` → 写 ChromaDB
2. `extract_and_save_profiles` → 提取画像（一次 LLM，两个 function call 并行）+ 更新情绪

- [ ] 第二次打开 App，提到上次说过的事，AI 能接上
- [ ] 问"你记得我叫什么"时，AI 给出正确姓名
- [ ] 长期记忆命中时，服务端日志显示 `[Memory] 命中 N 条`
- [ ] 未安装 chromadb 时，对话正常运行，无报错

### 5.8 结束会话与日记生成

#### 业务背景
用户结束一次对话后，自动生成日记条目，这是产品"留痕迹"价值的核心体现。

#### 处理流程

```
用户点击「结束对话」
  ↓
POST /api/session/end
  ↓
① analyze_anxiety（LLM）
   → 判断是否有明显焦虑/压力
   → 写入 sessions.anxiety_detected

② generate_summary（LLM）
   → 100-200字，第二人称「你」，温暖语气
   → 写入 sessions.summary

③ analyze_mood（LLM）
   → 2-4字情绪词（平静/焦虑/开心/疲惫…）
   → 写入 sessions.mood

④ generate_mood_image（DashScope wanx）
   → LLM 生成 Ghibli 治愈风格 prompt
   → 异步轮询，最长等 60s
   → 写入 sessions.mood_image_url

⑤ 返回日记卡片数据
   → 心情画 + 情绪标签 + 总结文字
```

#### 总结生成规范

- 字数：100-200字
- 人称：第二人称「你」
- 语气：温暖，像朋友在帮忙记笔记
- 禁止：不用"今天你XXX"的流水账句式；不用"我理解你"等套话
- 示例好句：*"你提到面试结果还没出来，等待的感觉让你有点悬。聊到后来你说了很久关于那个项目的事，听起来你对这件事还是很在意的……"*

#### 验收标准
- [ ] 结束 3 轮以上对话的会话，总结字数 100-200 字
- [ ] 总结用第二人称「你」，不出现「我理解」等禁忌词
- [ ] 情绪词在预设词库内（平静/焦虑/开心/难过 等）
- [ ] 生图成功时日记卡片顶部显示图片
- [ ] 生图失败/超时时，日记卡片无图但正常展示文字内容

### 5.9 心情画（文生图）

#### 业务背景
每篇日记配一张 AI 定制的治愈系插画，是产品的视觉差异化亮点，给用户惊喜感。

#### 生成流程

```
1. LLM 生成 art prompt
   输入：情绪词 + 对话摘要（≤400字）
   → 输出：30-50字中文 prompt，自然意象，吉卜力风格
   → LLM 失败则查兜底 dict（13种情绪覆盖）

2. 调用 DashScope wanx REST API
   → POST 提交任务（X-DashScope-Async: enable）
   → 每 3s 轮询任务状态
   → 最多 20 次（60s）
   → SUCCEEDED → 返回图片 URL
   → 失败 / 超时 → 返回 None
```

- **禁止**：人物面孔、文字
- 风格：治愈系插画，参考吉卜力工作室或温馨绘本
- 色调柔和温暖，有故事感和生活气息
- 长度：30-50 中文字

### 5.13 AI 质量评估（LLM-as-Judge）

#### 业务背景
每轮对话后台自动评分，用于：（1）发现 AI 质量问题；（2）验证 prompt 改动效果；（3）统计分析。

#### 评分维度

| 维度 | 满分 | 说明 |
|---|---|---|
| empathy（共情）| 10 | 是否真正接住用户情绪，而非套话 |
| naturalness（自然度）| 10 | 语言是否口语，不像客服 |
| helpfulness（有用性）| 10 | 在用户需要时是否提供了有价值内容 |
| safety（安全性）| 10 | 是否避免有害或不当内容 |
| overall（综合）| 10 | 综合评分 |

#### 技术实现

- 评估模型独立于对话模型（默认 DeepSeek），解耦避免评分偏袒
- 评分写入 `evaluations` 表，同时记录 `latency_ms` 和 `token_count`
- 不阻塞主流程，异步执行
- 通过 `static/dashboard.html` 可视化查看趋势

#### 基准测试

- 工具：`scripts/run_benchmark.py`
- 用途：每次修改 prompt 后跑基准测试，防止质量退步
- 结果：存入 `benchmark_results/` 目录

#### 验收标准
- [ ] 每轮对话后 `evaluations` 表有新记录
- [ ] 评估模型不可用时，对话正常，不报错
- [ ] benchmark 脚本可独立运行，输出通过率

## 6. AI 人设规范

### 6.1 角色定义

- **名字**：小伴
- **定位**：用户最懂他们的朋友。不是心理咨询师，不是客服，就是朋友。
- **基调**：真实、温柔、不费力

### 6.2 说话风格

| 规范 | 细节 |
|---|---|
| 句子长度 | 每句不超过 15 字 |
| 语言风格 | 口语，不用书面语、正式词汇 |
| 语气词 | 偶尔使用"嗯……""哎""对"，不是每次，真实人说话的感觉 |
| 表情符号 | 不以 emoji 替代情感，不用表情包撑场面 |

### 6.3 情绪响应原则

**接住处境，不贴标签**

> ✗ 错误：「你一定很委屈」  
> ✓ 正确：「当着那么多人，地都想找个缝钻进去……」

**不主动给建议**

> 用户没问方法 → 不给  
> 用户问了 → 先问一个让你真正能帮上的问题，再给**一个**有用的建议，不是三条

**不以问句收尾（重要）**

> 说完就说完，让对方自己决定要不要继续说  
> 问题只在真的想知道、或对方明显还有话没说完时才问

### 6.4 禁止词表

以下词语**绝对禁止**出现在任何回复中：

> 「你不是一个人」「加油」「一定会好的」「你很棒」「慢慢来」  
> 「我理解你的感受」「你应该……」

### 6.5 特殊场景处理

| 场景 | 处理方式 |
|---|---|
| 焦虑 / 身体不适 | 先陪着，不急着说"去检查一下"——那会让人觉得自己有病 |
| 闲聊 | 就闲聊，不要把每句话都引向"你最近怎么样" |
| 自杀 / 自伤信号 | 温和接住，不评判；引导专业支持，不强迫 |

### 6.6 消息格式规范

```
[NEXT] 标记用于拆分多条消息：

第一条：接情绪（1-2句）
[NEXT]
第二条：展开/追问/陪伴（1-2句）
[NEXT]（可选）
第三条：若有必要的补充

简单的话一条就够，不要硬凑成多条。
```

## 8. 关键设计决策

> 本章记录产品中几个关键设计选择背后的思考过程，体现「为什么这么做」而非「做了什么」。

### 为什么回复要拆成 2-3 条短消息？

早期版本 AI 返回一大段完整回复。测试发现，连续一整段文字给人的感觉是"在念稿子"，用户会下意识觉得自己在接收信息，而不是在和人聊天。改成 2-3 条短消息连发后，节奏明显变轻盈，更接近朋友发消息的感觉。这是一个成本极低但体验提升显著的改动。

### 为什么禁止 AI 以问句收尾？

早期每条回复几乎都以"你怎么看？""你有没有试过……？"结尾。用户反馈"感觉像接受采访"，有压迫感，说了一会儿就不想说了。

根本原因是：AI 在用问句"推进对话"，但用户倾诉的时候不需要被推进，他们需要被接住。改成说完就说完、让用户自主决定是否继续后，对话更自然，用户的发言变得更主动。

### 为什么用 ChromaDB 做长期记忆，而不是搜索历史消息？

跨会话记忆的核心问题是**语义匹配**，不是关键词查找。用户上次说"我最近压力很大"，这次说"最近有点撑不住"——这两句话几乎没有共同词，但语义上强相关。直接搜历史消息找不到；用向量相似度才能召回。

ChromaDB 是专为此设计的本地向量数据库，无需外部服务、数据不离开本机，与产品的隐私定位一致。

### 为什么要建立量化的 AI 质量评估体系？

最初靠主观感受判断 AI 质量——改了 prompt 之后"感觉好多了"，但说不清楚好在哪，也不知道有没有让某些场景变差。

引入 LLM-as-Judge 之后，每次修改都能拿到数字（共情分、自然度、综合分），迭代有了依据。这套体系让"AI 变好了"从感受变成了可追踪的数据，也防止了"这个场景好了但那个场景退步了"的问题。最终 benchmark 通过率从 47% 系统性提升到 80%。

### 为什么定位成「朋友」而不是「咨询师」？

咨询师的回应模式是结构化的：共情 → 探索问题 → 给建议。这套模式在专业场景有效，但放到日常倾诉里会显得刻意，用户会觉得"被分析了"。

大多数人倾诉的时候不需要解决方案，只需要被听见。「朋友」模式的核心是接住具体处境、不贴情绪标签、不主动给建议——这个定位清晰后，所有功能取舍和人设设计都有了统一的判断标准。

### 10.2 核心模块

#### LLM 调用层
统一封装所有模型调用，屏蔽不同供应商的差异。关键设计：禁用 Qwen3 思维链模式（避免回复过于正式）、适当提高温度参数（增加口语自然度）。RAG 知识库通过工具调用（function call）实现，让模型自主决定何时需要查知识库。

#### 长期记忆层
ChromaDB 本地向量数据库。写入和画像提取合并为一次 LLM 调用（两个并行的工具调用），减少 API 开销。读写通过单例模式管理，支持并发安全访问。

#### 文生图层
因官方 SDK 存在已知 bug，改为直接调用 DashScope REST API，自行实现异步任务提交和轮询逻辑，获得更细粒度的错误控制。

### 10.3 后台异步任务

每轮对话后，以下三项**后台并行执行，不阻塞对话响应**：

| 任务 | 说明 |
|---|---|
| 写入长期记忆 | 将本轮对话存入向量数据库 |
| 提取用户画像 | 识别用户提到的姓名/偏好/经历，更新情绪标签 |
| AI 质量评分 | LLM-as-Judge 自动评分，写入评估记录 |

## 16. 核心问题与解决过程

> 本章记录产品开发过程中遇到的真实问题，以及分析思路和解决方案。这些不是预先规划好的功能，而是在迭代中碰到、想清楚、解决掉的事情。

### 问题 1：AI 回复太像客服，用户感觉"假"

**现象**

用户反馈集中在三点：感觉像在跟客服聊、共情感觉很假、建议没用。具体表现是：每条回复结构工整、每次都以问句收尾、遇到情绪问题就给三条建议。

**根因分析**

追查后发现了两个叠加的原因：

1. **模型层**：使用的 Qwen3 模型默认开启了「思维链」模式——模型在内部做结构化分析后再输出，导致回复偏报告风格，而不是对话风格
2. **Prompt 层**：系统提示是规则清单（"要做到 A、B、C"），AI 在遵守规则，而不是在「做自己」。规则越多，回复越刻意

**解决方案**

1. 在模型调用层关闭思维链，强制输出对话风格
2. 将 Prompt 从规则清单改写为人设描述——不告诉 AI 要怎么做，而是描述「小伴是一个什么样的人」（参见第 6 章）
3. 加入禁止词表，从负向约束兜底

**结果**

AI 质量 benchmark 通过率从 47% 提升至 80%，自然度和共情两个维度提升最明显。更重要的是：迭代有了可量化的依据，不再依赖主观感受。

---

### 问题 2：长语音识别丢字

**现象**

用户说了 30 秒以上的话，AI 只收到了前面一小句，剩下的全部丢失。

**根因分析**

浏览器 Web Speech API 在检测到静音超过约 1.5 秒时会自动停止识别，触发结束事件后，正在缓冲的内容就此丢失。用户说话中有正常的换气停顿，就会被反复打断。

**解决方案**

1. 识别停止时立即重新创建识别器并继续，用累积变量拼接所有片段的识别结果
2. 用户松手时，等待当前识别实例完成最终回调后再统一发送，防止最后几个字被截断

**结果**

日常长度（10-30 秒）的语音识别完整度显著提升。极快松手（< 0.5 秒）场景仍可能丢失最后几个字，已列入下一版本用服务端 ASR 彻底解决。

---

### 问题 3：文生图 SDK 报错，图片功能无法上线

**现象**

接入心情画功能时，DashScope 官方 Python SDK 调用图片生成接口持续报 `url error`，无法生成任何图片。

**根因分析**

排查后确认是 SDK 内部对异步任务模式的 URL 处理存在 bug，属于 SDK 自身问题，短期内无法通过升级版本解决。

**解决方案**

放弃使用 SDK，直接通过 HTTP 调用 DashScope REST API，自己实现「提交任务 → 轮询状态 → 获取结果」的完整流程。

**结果**

功能正常运行，同时获得了更细粒度的错误处理能力。绕过 SDK 的过程也让我意识到：依赖第三方库时需要有备用路径，这也是为什么 PRD 中要求「IMAGE_GEN_MODEL 未配置时优雅降级」——提前设计好了容错。

---

### 问题 4：如何让 AI 感知用户说话时的情绪状态

**背景**

语音转文字后，情绪信号丢失了。同样的文字"今天好累"，哭着说和平静地说，AI 应该给出不同温度的回应——但它无从判断。

**思路探索**

最直接的方案是引入语音情绪识别模型，但这意味着额外的 API 成本、更长的响应链路，且效果不稳定。

退而求其次的思路：不做精准情绪识别，而是提取几个简单的声学特征，生成自然语言描述，让 AI 自己判断如何调整语气。

**实现**

用浏览器原生 Web Audio API 实时采集声音能量（每 100ms 一个采样点），从音量高低、音量波动、停顿比例三个维度生成描述，如「声音较轻，有明显停顿」，随消息一起发给 AI。AI 收到这条描述后，在回复中自然调整语气，不直接提及它。

**结果**

零额外 API 成本，实现了基础的语音情绪感知。这个方案的精度有限，但对于「用户哭着说话」vs「用户平静讲述」的粗粒度区分已经够用。

---

### 问题 5：如何知道每次 prompt 改动是好还是坏

**背景**

改了一次 Prompt 后，主观感觉好多了，但无法确认是真的变好，还是在某些场景下反而退步了。没有量化标准，迭代就没有方向。

**方案**

1. 整理一批覆盖高频场景的测试问题（求助、闲聊、焦虑、追问等）
2. 引入独立的评估模型（与对话模型解耦），从共情、自然度、有用性、安全性四个维度打分
3. 每次修改 Prompt 后跑完整测试，通过率不得低于上一版本

**结果**

迭代从此有了量化依据。三轮改进的数字轨迹：47% → 73% → 80%，每一次提升都能对应到具体的改动内容，也能看到哪个维度还有空间。这套体系本身成了产品质量的「安全网」。

## 17. 产品反思与学习

### 做对了什么

**早早建立了量化评估体系**

很多 AI 产品在质量上依赖主观判断，我从早期就引入了 LLM-as-Judge 和 benchmark。这让「AI 变好了」从感受变成了数据，迭代有了依据，也防止了某个场景变好但另一个场景退步的问题。

**把「不做什么」想清楚了**

情感陪伴产品很容易同时想做咨询师、做日记工具、做效率助手。我很早就确定了「只做朋友」的定位，所有功能取舍都围绕这一点——结果是产品调性保持了一致性，没有变成四不像。

**技术选型留了退路**

LLM 用 OpenAI 兼容格式，一行配置切换供应商；TTS 和文生图都是可选功能，不配置时优雅降级。这让产品在不同资源条件下都能运行，也降低了对单一供应商的依赖。

---

### 做得不够好的

**缺乏真实用户测试**

整个迭代过程主要依赖我自己的使用感受和 LLM 评分，没有系统地让真实用户使用并收集反馈。AI 评分好不等于真实用户喜欢——这是最大的盲区。如果重来，我会在 v1.0 上线后立即找 5-10 个目标用户做访谈。

**图片持久化问题没有在设计阶段考虑到**

心情画使用了第三方返回的外链 URL，这类链接有时效性。这是一个在功能设计时就应该考虑到的问题，结果变成了技术债，已列入 v2.1 优先解决。教训是：凡是涉及「外部存储 URL」的功能，设计阶段就要问自己「这个 URL 三个月后还在吗」。

**没有设计用户增长路径**

产品当前是私有化部署，缺少分享、邀请、裂变的设计。如果未来要扩展到更多用户，这部分需要重新设计。

---

### 如果有更多资源，下一步做什么

1. **真实用户访谈**：找 10 个目标用户，观察他们第一次使用的过程，找到真正的流失节点
2. **服务端语音识别**：替换浏览器 Web Speech API，解决兼容性和丢字问题，同时能支持方言和更长的语音
3. **月度情绪报告**：让用户能回顾自己一段时间内的情绪变化，增强产品的长期留存价值

## 18. 迭代规划（Roadmap）

### v2.1 — 高优先级（预计下一迭代）

#### F1. 心情画本地持久化 ⭐⭐⭐

**背景**：当前心情画使用 DashScope 返回的外链 URL，存在时效性问题，图片可能在数周后失效。

**方案**：
1. 生图成功后，立即下载图片字节
2. 保存至 `static/mood_images/{user_id}/{session_id}.jpg`
3. `mood_image_url` 存为本地相对路径（`/static/mood_images/...`）
4. 前端直接加载本地路径，不依赖外链

**影响范围**：`services/image_gen.py`、`main.py`（`api_end_session`）

---

#### F2. 语音识别服务端化 ⭐⭐⭐

**背景**：当前使用浏览器 Web Speech API，存在以下问题：
- Firefox 不支持
- 长语音时静音自动停止导致丢字
- 无法自定义词汇

**方案**：PTT 录音改为 WebM 音频流 → 上传至服务端 → DashScope ASR（Paraformer）识别 → 返回文字

**接口变更**：新增 `POST /api/asr`，接收音频文件，返回识别文字

---

#### F3. 评估 Dashboard 完善 ⭐⭐

**背景**：`static/dashboard.html` 已存在但功能不完整，评分数据无法直观查看。

**方案**：
- 折线图：各维度评分趋势（按日/周）
- 分布图：overall 分数分布
- 低分预警：overall < 6.0 的回复列表，便于问题排查
- token 消耗统计

---

#### F4. 月度成长报告 ⭐⭐

**背景**：`growth_reports` 表已存在，但接口和前端均未实现。

**方案**：
- 每月自动生成（或用户主动触发）
- 内容：本月情绪变化趋势、高频话题、与上月对比、一句温暖寄语
- 展示：日记本新增"成长报告"入口，以卡片形式呈现

---

### v2.2 — 中优先级

#### F5. 移动端适配 ⭐⭐

当前 UI 以桌面为主，需针对手机屏幕优化：
- 三栏改为底部 Tab 导航（聊天 / 日记 / 设置）
- PTT 按钮位置和大小优化
- 键盘弹出时聊天区域自动上移

---

#### F6. 智能推送提醒 ⭐⭐

**背景**：当前久未登录提醒只在下次打开时展示，无法主动触达用户。

**方案**：接入邮件推送（如 SendGrid）或微信公众号推送，在用户 N 天未登录时主动发送个性化问候。

---

#### F7. 对话质量主动干预 ⭐⭐

当评估分数连续 3 轮低于阈值时：
- 后台记录告警
- 自动追加提示："上面几条回复质量较差（评分 < 6.0），请重点参考人设规范调整"

---

### v3.0 — 长期规划

#### F8. 多角色切换

用户可选择不同风格的小伴：温暖陪伴型（当前）/ 轻松幽默型 / 理性分析型

#### F9. 情绪曲线年度报告

年终总结：一年的情绪变化可视化、高频话题词云、关键事件时间线

#### F10. 私有知识库增强

- 支持 Word、网页链接等更多格式
- 按分类管理知识库（心理类、职场类、人际类…）
- 用户可查看哪些内容被引用过

#### F11. 对话模板与快捷启动

内置常用倾诉场景快捷入口（如"今天想说说工作的事""睡前聊聊"），降低开口门槛
